# 00 项目定位说明

这一篇放在整个 `docs/code-reading` 的最前面，只回答一个问题：

- OSWorld 这个项目到底是在评测什么？

如果你先把这个问题想清楚，后面读代码时就不会总把它误解成：

- 纯 GUI 理解项目
- 纯桌面自动化工具
- 纯 agent 框架

它其实都沾一点，但又都不完全是。

## 先给一句最准确的总结

OSWorld 更准确的定位是：

- 一个 **面向桌面任务的 agent benchmark**

或者再展开一点：

- 一个把 `任务定义`、`桌面环境`、`agent 执行`、`结果评测` 组合在一起的桌面智能体评测框架

所以如果你问：

- “它是不是用来评测 agent 对 GUI 的理解能力？”

答案是：

- `是，但不只如此。`

## 一、它评测的不是单点 GUI 识别，而是整条任务链

这个项目最终评测的是：

- agent 能不能在真实桌面环境里，把一个自然语言任务做完

这里面至少包含 4 层能力。

### 1. GUI 感知

agent 要看懂当前界面。

例如：

- 当前窗口是什么
- 哪些元素可点击
- 表格和输入框在哪里
- 页面是不是已经跳转

这部分通常来自：

- `screenshot`
- `accessibility_tree`

所以说它确实覆盖了：

- GUI understanding

### 2. 任务理解与规划

agent 不只是“看懂界面”，还要把自然语言 instruction 变成行动计划。

例如：

- 先打开设置
- 再找到某个选项
- 再修改并保存
- 最后判断是否可以结束

这部分更像：

- task planning

### 3. 动作执行

agent 还要真的把动作落到桌面里。

例如：

- 鼠标点击
- 键盘输入
- 快捷键
- 拖拽
- 在 Chrome、Calc、GIMP 等应用里真实操作

这部分不是“想到了就算”，而是必须：

- `env.step(action)` 真正执行成功

### 4. 结果达成

最后还要验证任务是否真的完成。

这一步不是靠模型自己说“我做完了”，而是靠 evaluator 去检查：

- 配置是否真的被改掉
- 文件是否真的保存正确
- 表格内容是否正确
- 页面状态是否符合预期

所以项目最终评的是：

- `任务完成度`

而不只是：

- `界面理解得像不像`

## 二、为什么不能把它简单叫做“GUI 理解 benchmark”

因为“GUI 理解 benchmark”通常更像在测：

- 图像里识别按钮
- 找控件
- 定位元素
- 做 grounding

但 OSWorld 明显比这个更重。

它不是只问：

- “这个按钮在哪？”

而是问：

- “你能不能利用当前桌面环境，把整件事做完？”

举几个例子：

- 在 Chrome 里把默认搜索引擎改成 Bing
- 在 Calc 里补公式、导出文件、保留格式
- 在 GIMP 里做图像处理

这些任务都要求 agent 同时具备：

- 理解界面
- 操作应用
- 维持上下文
- 达成最终结果

所以它比纯 GUI 理解更接近：

- `desktop task execution benchmark`

## 三、为什么它也不能简单叫“桌面自动化工具”

因为它的目标不是替你稳定执行一套固定脚本。

桌面自动化工具更像：

- RPA
- 录制回放
- 预先写好的脚本

而 OSWorld 的设计是：

- 给 agent 一个任务
- 给它一个环境
- 让它自己决定怎么完成

也就是说，它更强调：

- agent 的泛化能力

而不是：

- 固定流程自动化

## 四、为什么它也不只是“agent 框架”

它当然有 agent 代码，例如：

- `mm_agents/`

但这个仓库的重点不只是“怎么写一个 agent”，还包括：

- 如何准备任务
- 如何提供桌面环境
- 如何执行动作
- 如何设计 evaluator
- 如何批量跑 benchmark

所以如果只把它看成：

- 一个多模态 agent 实现仓库

也会低估它。

更准确地说，它是：

- `agent + environment + benchmark + evaluator`

一起组成的系统。

## 五、这个项目最核心的评测对象是什么

如果要再压缩成一句话，我建议你这样记：

- 它评测的是 **agent 在真实桌面操作任务上的端到端能力**

这里的“端到端”很重要。

所谓端到端，不是指神经网络结构，而是指任务链条完整：

1. 收到自然语言目标
2. 获取桌面观测
3. 生成动作
4. 操作真实应用
5. 修改真实状态
6. 被 evaluator 验证结果

这条链里任何一环掉了，最终都不会得分。

## 六、为什么 evaluator 是理解项目定位的关键

很多人第一次看这类项目，容易把重点全放在：

- 模型怎么读 screenshot
- prompt 怎么写
- action space 长什么样

这些当然重要，但如果你想真正理解 OSWorld，必须看 evaluator。

因为 evaluator 决定了：

- 项目到底认为“成功”是什么

而 OSWorld 里的“成功”通常不是：

- 看起来像完成

而是：

- 配置文件真的改对了
- 导出的文件真的对了
- 表格显示结果真的对了
- 某个应用状态真的落盘了

所以这个项目的本质更偏：

- 任务结果评测

而不只是：

- 行为轨迹展示

## 七、你可以把它和哪些东西区分开

为了避免后面继续混淆，建议你主动把它和三类东西区分开。

### 1. 和纯视觉 grounding 数据集区分

纯视觉 grounding 更关注：

- 点哪里
- 框哪里
- 元素识别是否准确

OSWorld 更关注：

- 整个任务有没有做成

### 2. 和 RPA / workflow automation 区分

RPA 更关注：

- 稳定执行确定流程

OSWorld 更关注：

- 在开放任务里自主决策

### 3. 和纯 LLM agent 框架区分

很多 agent 框架更关注：

- 工具调用
- memory
- planning

OSWorld 更关注：

- 在真实 GUI 环境里的 benchmark 结果

## 八、如果带着这个定位去读代码，主线会更清楚

你后面读代码时，最好一直用下面这条主线去看：

`task json -> env.reset -> agent.predict -> env.step -> env.evaluate`

这条链分别对应：

1. `evaluation_examples/`
   定义任务
2. `desktop_env/`
   提供环境和执行
3. `mm_agents/`
   负责决策
4. `evaluator`
   判断结果

只要你不丢这条主线，就不会把项目误看成单独某一层。

## 九、如果只用一句最短的话介绍这个项目

你以后如果要给别人介绍 OSWorld，可以直接说：

- 这是一个评测桌面 agent 端到端任务完成能力的 benchmark 项目。

如果对方再问一句：

- “是不是也在测 GUI 理解？”

你就补一句：

- 是，但 GUI 理解只是其中一层，它最终测的是任务完成。
